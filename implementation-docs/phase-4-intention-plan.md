# Phase
Phase 4

# Summary
Phase 4 changes only the existing root-page frontend presentation layer so the recommended next step becomes the first visible interaction when available, task creation becomes the first visible interaction only when no recommendation or active session exists, copy shifts from phase wording to activation wording, loading and error language becomes action-specific, and the existing page gains bounded motion plus tighter responsive behavior. The implementation is strictly limited to the six approved frontend files already present in the repository and must not alter backend contracts, frontend API methods, shared type files, or add any new file.

- Existing backend surface: The backend already exposes the phase 4 data and actions through the existing FastAPI routes under `backend/app/api/routes/`. The phase 4 plan explicitly forbids backend edits, and the current repository already contains the required route files, schema files, domain services, and repositories. No backend file is a safe write target in this phase.
- Existing frontend surface: The root page already exists at `frontend/app/page.tsx` and already owns subjects, tasks, active session, active instruction, next task, pending state, and error state. - Existing tests: No product test surface exists in the current repository tree. There is no existing `backend/tests`, `frontend/__tests__`, `frontend/tests`, or equivalent application-test directory to name as a modification target. Only documentation test-report artifacts exist under `implementation-docs/`.
- Missing required surfaces: No new surface is required. The approved phase 4 plan is modifications only. The repository already contains every implementation file that phase 4 is allowed to touch.

# Required Contracts Between Files
- `frontend/app/page.tsx` remains the only frontend file allowed to own network state, pending state, error state, recommendation state, and action sequencing. `frontend/components/task-form.tsx`, `frontend/components/task-list.tsx`, and `frontend/components/session-view.tsx` remain presentational only.
- `frontend/app/page.tsx` must continue using the already-existing API client methods imported from `frontend/services/api.ts`. Phase 4 does not authorize any new client method, request shape, response shape, or fetch helper change.
- `frontend/app/page.tsx` may add only the presentation-level `pendingAction` distinction on top of the current `isPending` flow. It must not change when create, start, complete, or abort requests occur; it may only change the displayed wording and the DOM order of existing panels.
- `frontend/components/task-form.tsx` must continue receiving the raw title value, subject list, selected subject id, disabled state, and callbacks from `frontend/app/page.tsx`. It must not own validation logic beyond honoring the existing disabled conditions already computed in the component.
- `frontend/components/task-list.tsx` may receive one recommendation-aware wording signal from `frontend/app/page.tsx`, but it must not receive or derive business rules that change task ordering or recommendation selection.
- `frontend/components/session-view.tsx` must continue receiving already-resolved `session`, `instruction`, `nextTask`, and `nextTaskSubject` props from `frontend/app/page.tsx`. It must not fetch, match subject ids, or interpret backend state beyond choosing among its existing three render branches.
- `frontend/app/globals.css` may style only the existing page and component structure plus any minimal new class hooks needed for the approved copy and motion. It must not introduce styles for new routes, new overlays, or new component files.

# Risk Points
- The highest implementation-drift risk is widening scope into `frontend/services/api.ts` or `frontend/types/study-buddy.ts`. The current repository already has the required data flow for phase 4, and the approved plan forbids those edits.
- DOM-order drift is a concrete risk in `frontend/app/page.tsx`: the session panel must become the first interactive block when a session or recommendation exists, not merely appear visually earlier through CSS-only reordering while remaining later in the DOM.
- State-flow drift is a concrete risk in `frontend/app/page.tsx`: `pendingAction` is allowed only to drive wording and status presentation. It must not change the underlying create, start, complete, or abort request sequence already present.
- Frontend-business-logic drift is a concrete risk in `frontend/components/task-list.tsx` and `frontend/components/session-view.tsx`: recommendation-aware wording is allowed, but recommendation selection, task ranking, and any backend-owned decision logic must not move into these components.
- Accessibility drift is a concrete CSS risk: new focus styling must use `:focus-visible` and existing accent tokens rather than replacing the control system with unrelated custom interaction patterns.
- Responsive drift is a concrete CSS risk: the approved breakpoints are exact and must all exist. Reusing only the current 900px and 640px rules is insufficient because phase 4 explicitly adds 1024px and 480px behaviors.
