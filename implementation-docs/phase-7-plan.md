## Phase
Phase 7: System Contracts and Enforcement

## Phase Goal
Phase 7 defines the executable backend contract for the existing Study Buddy flow by enforcing request DTO validation at the FastAPI boundary, exposing a machine-readable API contract through the running application, adding process liveness and dependency readiness endpoints, normalising backend error responses into one documented shape, and proving the phase with a runnable start-to-finish user flow plus at least one deliberate contract violation that fails immediately and clearly.

## Scope
### Included
- Backend API-edge validation for request bodies and route parameters on the existing `/tasks`, `/subjects`, `/next`, and `/sessions` route surface.
- A shared backend error response schema with one JSON shape used by handled validation, domain, readiness, and unexpected-failure responses.
- Explicit FastAPI contract metadata for the existing business routes plus `GET /health` and `GET /ready`.
- A machine-readable OpenAPI contract surface served by the running backend and verified by automated tests.
- `GET /health` for process liveness with no database dependency.
- `GET /ready` for database-pool and database-query readiness using the existing backend connection layer.
- Automated proof that malformed input fails at the boundary before business logic runs.
- Automated proof that an invalid response payload fails as a contract violation instead of returning a silent partial success.
- An explicit runnable user flow definition covering load subjects, create task, start task, complete session, and refresh next task through the existing frontend-backed route sequence.

### Excluded
- Any new product feature, new study behavior, new recommendation rule, new personalization rule, or new subject behavior.
- Any database schema change, seed change, migration, index, trigger, or data backfill.
- Any change to domain-engine formulas, task-ranking logic, subtask-generation rules, difficulty-reduction rules, or session-resolution rules.
- Any direct frontend validation, frontend readiness polling, frontend health UI, frontend OpenAPI consumption, or frontend business-logic addition.
- Any new authentication, authorization, rate limiting, analytics, metrics dashboard, tracing pipeline, or alerting system.
- Any runtime-config validation work from Phase 8, including environment-schema enforcement across frontend and backend.
- Any new dependency addition or replacement of FastAPI, Pydantic, asyncpg, or the existing pytest-based backend test surface.

## Preconditions
- `implementation-docs/devplan.md` defines Phase 7 as API-edge validation, machine-readable contracts, health and readiness endpoints, consistent backend errors, and runnable-flow completion proof.
- The backend already runs on FastAPI in `backend/app/main.py` and already exposes the existing business routers from `backend/app/api/routes/tasks.py`, `backend/app/api/routes/sessions.py`, and `backend/app/api/routes/subjects.py`.
- The backend already uses Pydantic request and response models in `backend/app/api/schemas/tasks.py`, `backend/app/api/schemas/sessions.py`, and `backend/app/api/schemas/subjects.py`, but blank-title enforcement and malformed path-parameter enforcement are not yet fully bounded to the FastAPI boundary.
- The backend already uses pooled PostgreSQL access through `backend/app/infrastructure/db/connection.py`, so readiness checks can reuse the existing pool instead of adding a new infrastructure component.
- The frontend root flow in `frontend/app/page.tsx` already executes the user path that this phase must prove: `GET /subjects`, `GET /tasks`, `GET /next`, `POST /tasks`, `POST /tasks/{task_id}/start`, `POST /sessions/{session_id}/complete`, and `POST /sessions/{session_id}/abort`.
- The frontend API client in `frontend/services/api.ts` already reads backend errors from a `detail` field, so the phase error contract must preserve `detail` as the client-visible error field.
- The repository contains no `.github/workflows` directory, so CI proof is not wired today.

## Backend Design
### Routes
### OpenAPI Contract Surface
- Method: GET
- Path: /openapi.json
- Request Shape: No request body and no route parameters.
- Response Shape: OpenAPI JSON generated by the FastAPI application and containing the existing business routes plus `GET /health` and `GET /ready`, their success schemas, and the shared error schema for documented failure statuses.
- Error Conditions: `500` when OpenAPI generation fails.

### Health
- Method: GET
- Path: /health
- Request Shape: No request body and no route parameters.
- Response Shape: JSON object with `status` set to `ok`.
- Error Conditions: No dependency failure path is allowed on this route. `500` is reserved for unexpected application failure.

### Readiness
- Method: GET
- Path: /ready
- Request Shape: No request body and no route parameters.
- Response Shape: JSON object with `status` set to `ready` and `database` set to `ok`.
- Error Conditions: `503` when the asyncpg pool is unavailable, connection acquisition fails, or the readiness probe query fails. `500` for unexpected application failure.

### Create Task
- Method: POST
- Path: /tasks
- Request Shape: JSON object with `title` as a required string that is trimmed at the boundary and rejected when blank after trimming, and `subject_id` as an optional UUID.
- Response Shape: Unchanged task object with `id`, `title`, `subject_id`, `created_at`, and `is_completed`.
- Error Conditions: `400` when `subject_id` is well-formed but violates the subject foreign-key contract. `422` when `title` is missing, blank after trimming, or `subject_id` is malformed. `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### List Tasks
- Method: GET
- Path: /tasks
- Request Shape: No request body and no route parameters.
- Response Shape: Unchanged JSON array of task objects with `id`, `title`, `subject_id`, `created_at`, and `is_completed`.
- Error Conditions: `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### Start Task
- Method: POST
- Path: /tasks/{task_id}/start
- Request Shape: Route parameter `task_id` parsed as a UUID by the FastAPI boundary. No request body.
- Response Shape: Unchanged `TaskStartResponse` object with `session` and `instruction`.
- Error Conditions: `404` when the task does not exist. `409` when the task is already completed or already has an open session. `422` when `task_id` is not a valid UUID. `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### List Subjects
- Method: GET
- Path: /subjects
- Request Shape: No request body and no route parameters.
- Response Shape: Unchanged JSON array of subject objects with `id`, `name`, `color`, and `icon`.
- Error Conditions: `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### Get Next Task
- Method: GET
- Path: /next
- Request Shape: No request body and no route parameters.
- Response Shape: Unchanged task object with `id`, `title`, `subject_id`, `created_at`, and `is_completed`.
- Error Conditions: `404` when no incomplete task exists. `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### Complete Session
- Method: POST
- Path: /sessions/{session_id}/complete
- Request Shape: Route parameter `session_id` parsed as a UUID by the FastAPI boundary. No request body.
- Response Shape: Unchanged session object with `id`, `task_id`, `subtask_id`, `started_at`, `ended_at`, `planned_duration_minutes`, `actual_duration_minutes`, `was_completed`, and `was_aborted`.
- Error Conditions: `404` when the session does not exist. `409` when the session is already ended. `422` when `session_id` is not a valid UUID. `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### Abort Session
- Method: POST
- Path: /sessions/{session_id}/abort
- Request Shape: Route parameter `session_id` parsed as a UUID by the FastAPI boundary. No request body.
- Response Shape: Unchanged session object with `id`, `task_id`, `subtask_id`, `started_at`, `ended_at`, `planned_duration_minutes`, `actual_duration_minutes`, `was_completed`, and `was_aborted`.
- Error Conditions: `404` when the session does not exist. `409` when the session is already ended. `422` when `session_id` is not a valid UUID. `503` when the database is unavailable. `500` for response-contract violation or unexpected internal failure.

### Domain Logic
- `backend/app/api/schemas/tasks.py` owns request-boundary enforcement for task creation. It trims `title`, rejects a blank post-trim value, and preserves the existing success DTO fields for `TaskResponse`, `InstructionResponse`, `TaskSessionResponse`, and `TaskStartResponse`.
- `backend/app/api/schemas/sessions.py` and `backend/app/api/schemas/subjects.py` remain the source of truth for backend success-response DTOs for session and subject payloads.
- `backend/app/api/schemas/errors.py` is added as the single backend error DTO. Every handled backend failure documented in OpenAPI uses this schema and preserves `detail` as the client-visible field.
- `backend/app/api/schemas/system.py` is added for the `GET /health` and `GET /ready` success payloads so those routes are contract-bound in the same way as the business routes.
- `backend/app/api/routes/tasks.py`, `backend/app/api/routes/sessions.py`, and `backend/app/api/routes/subjects.py` remain thin route modules. They accept typed boundary inputs, call existing services, and declare the shared error response schema for their non-2xx statuses.
- `backend/app/api/routes/system.py` is added as the only new route module in this phase. It exposes `GET /health` and `GET /ready`, and it delegates dependency probing to the connection layer instead of embedding database access logic in the route.
- `backend/app/main.py` owns shared exception handling for `RequestValidationError`, `ResponseValidationError`, `HTTPException`, `DatabaseUnavailableError`, and uncaught exceptions so every handled failure returns one JSON error shape and one predictable status code.
- `backend/app/main.py` also owns router registration for the system routes and the running OpenAPI surface that documents every route contract in this phase.
- `backend/app/infrastructure/db/connection.py` owns readiness probing. The probe reuses the existing pool, verifies that the pool exists, acquires one connection, runs one lightweight database query, and raises `DatabaseUnavailableError` on any readiness failure.
- `backend/app/domain/services/task_service.py`, `backend/app/domain/services/session_manager.py`, and `backend/app/domain/services/subject_service.py` keep their existing decision rules. Phase 7 does not move validation or readiness logic into these services and does not change any business outcome path.

### Persistence
- No table definition changes occur in `subjects`, `tasks`, `subtasks`, `sessions`, or `attempts`.
- No new column, index, migration, seed change, or persistence side effect is introduced in this phase.

- The existing create-task, list-task, next-task, start-task, complete-session, abort-session, and list-subjects persistence paths remain unchanged in purpose and write behavior.

## Implementation Sequence
1. Create `backend/app/api/schemas/errors.py` with the shared backend error DTO and create `backend/app/api/schemas/system.py` with the liveness and readiness success DTOs.
2. Modify `backend/app/api/schemas/tasks.py` so task creation enforces the trimmed non-blank title rule at DTO parse time while preserving the existing success DTO field set for all task-start payloads.
3. Modify `backend/app/api/routes/tasks.py`, `backend/app/api/routes/sessions.py`, and `backend/app/api/routes/subjects.py` so every route declares the shared error schema for its non-2xx statuses and uses typed UUID route parameters where the contract requires UUID input.
4. Modify `backend/app/infrastructure/db/connection.py` to expose one readiness probe that fails with `DatabaseUnavailableError` when the pool is absent, acquisition fails, or the probe query fails.
5. Create `backend/app/api/routes/system.py` with `GET /health` and `GET /ready`, keeping `GET /health` process-only and routing `GET /ready` through the connection-layer readiness probe.
6. Modify `backend/app/main.py` to register the system router, register shared exception handlers for request-validation, response-validation, database-unavailable, HTTP, and uncaught failures, and keep the FastAPI-generated OpenAPI surface enabled as the machine-readable contract endpoint.
7. Create `backend/tests/test_system_routes.py` to prove `/health` returns its liveness contract without touching the database and `/ready` returns either the readiness success contract or the documented `503` error contract.
8. Create `backend/tests/test_api_contract_surface.py` to prove `/openapi.json` contains the business routes plus `/health` and `/ready`, to prove invalid UUID and invalid task-create payloads fail with the documented `422` error shape, and to prove at least one deliberate invalid response payload becomes a `500` contract failure.
9. Modify `backend/tests/test_task_routes.py`, `backend/tests/test_session_routes.py`, and `backend/tests/test_subject_routes.py` so existing route tests stay aligned with typed path inputs, DTO-bound validation, and the shared error contract.
10. Modify `backend/tests/test_connection.py` so readiness-probe success and failure paths are covered alongside the existing pool lifecycle coverage.
11. Run the backend pytest suite and confirm that the only code-surface expansion in this phase is the shared error DTOs, the system route DTOs, the system route module, and the contract-oriented tests listed in this plan.

## Completion Criteria
1. `GET /health` returns `200` with the documented liveness payload and does not fail because the database is unavailable.
2. `GET /ready` returns `200` with the documented readiness payload only when the asyncpg pool exists, connection acquisition succeeds, and the readiness probe query succeeds.
3. `GET /ready` returns `503` with the shared backend error shape when the pool is absent, the database is unreachable, or the probe query fails.
4. `POST /tasks` rejects a missing title, a blank-after-trim title, or a malformed `subject_id` at the FastAPI boundary with `422` and the shared backend error shape.
5. `POST /tasks/{task_id}/start`, `POST /sessions/{session_id}/complete`, and `POST /sessions/{session_id}/abort` reject malformed UUID route parameters at the FastAPI boundary with `422` and the shared backend error shape.
6. Existing domain failures remain mapped to their current business statuses: `404` for not found, `409` for start or session state conflicts, `400` for subject foreign-key failure on task creation, and `503` for database unavailability.
7. At least one deliberate invalid response payload from a route test path fails as a `500` contract violation instead of returning a malformed success response.
8. `/openapi.json` documents the existing business routes plus `/health` and `/ready`, and documents the shared backend error schema for the statuses defined in this plan.
9. The runnable user flow remains intact with unchanged success payload shapes for `GET /subjects`, `POST /tasks`, `POST /tasks/{task_id}/start`, `POST /sessions/{session_id}/complete`, and `GET /next`.
10. No frontend file, domain-service file, domain-engine file, repository file, database schema object, or dependency file changes in Phase 7.

## Fail-Fast Mechanisms
- `CreateTaskRequest` fails request parsing when `title` is blank after trimming, so blank task titles do not reach route logic or service logic.
- Typed UUID route parameters fail request parsing for malformed `task_id` and `session_id`, so malformed identifiers do not reach service logic.
- Shared request-validation handling converts FastAPI boundary validation failures into one documented `422` error shape.
- Shared response-validation handling converts invalid route success payloads into one documented `500` contract-failure shape.
- The readiness probe fails immediately with `503` when the pool is missing or the probe query fails, and it never returns a false `200` readiness state.
- Contract-surface tests fail the phase when `/openapi.json` omits a required route, omits a required status contract, or drifts from the shared error schema.
